# Backend Internals

## Overview

The Viva Napoli backend is a high‑performance REST API written in **Go 1.22**. It follows idiomatic Go conventions, emphasizing simplicity, type safety, and clear separation of concerns. This document describes the internal architecture, core patterns, and development practices.

## Project Structure


## Core Patterns

### Handler as Dependency Container

The `Handler` struct (`internal/handler/handler.go`) serves as a **dependency injection container** that holds all shared resources:

```go
type Handler struct {
    queries *generated.Queries // Type‑safe SQL interfaces from sqlc
    config  *config.Config     // Strongly‑typed environment configuration
    resend  *resend.Client     // Email service client (Resend)
}
```

This pattern ensures that:

- Dependencies are initialized once and shared across all requests.
- Handlers are easy to test (dependencies can be mocked).
- The application’s lifecycle is clear (dependencies are created in `main.go` and passed to `Handler.New`).

### Request Validation

Every POST/PUT request defines a dedicated **request struct** with a `validate()` method. For example, `createOrderRequest` (`internal/handler/orders.go`) validates customer name, phone format, order type, and items.

Validation is performed **before** any database interaction, ensuring data integrity and providing clear error messages to the client.

### Price Snapshotting

To protect against price fluctuations and prevent tampering, the backend implements **price snapshotting** during order creation:

1. **Fetch Current Data**: The handler retrieves the current price, name, and availability of each menu item from the database using the IDs sent by the frontend.
2. **Recalculate Totals**: The total price is recalculated server‑side using the fetched prices. **Frontend‑provided prices are ignored.**
3. **Store Snapshot**: A JSON snapshot of each item (name, quantity, size, unit price, total price) is stored in the `orders.items` column (PostgreSQL `jsonb`). This snapshot preserves the exact state of the menu at the time of purchase.

This approach guarantees that:

- Historical orders remain accurate even if menu items are later updated or deleted.
- Customers cannot manipulate prices by modifying client‑side code.

## External Integrations

### Resend (Transactional Email)

When a new order is successfully placed, the backend sends an email notification to the restaurant owner using the **Resend** API.

- **Asynchronous Dispatch**: The email is sent in a separate goroutine after the database transaction is committed. This prevents email delays from affecting the HTTP response time.
- **Resilience**: If the Resend API call fails, the error is logged but the order is **not** rolled back. The restaurant can still see the order in the admin dashboard.
- **Dynamic Templates**: The email content is generated as HTML within `orders.go`, using Go’s `html/template` package for safety.

## Database Interaction with sqlc

Instead of a traditional ORM, we use **sqlc** to generate type‑safe Go code from SQL queries.

### Workflow

1. **Write SQL**: Define queries in `internal/db/queries/*.sql` (e.g., `menu_items.sql`).
2. **Configure sqlc**: The `sqlc.yaml` file specifies the PostgreSQL schema and output paths.
3. **Generate Code**: Run `make sqlc` to produce Go interfaces in `internal/db/generated/`.
4. **Use in Handlers**: The generated `Queries` struct provides methods like `GetAvailableMenuItemsByCategory` that accept a `context.Context` and return strongly‑typed results.

### Transactions

Complex operations (e.g., order creation that involves multiple queries) are wrapped in `pgx` transactions:

```go
tx, err := h.queries.BeginTx(ctx, pgx.TxOptions{})
// ... perform queries using tx
err = tx.Commit(ctx)
```

This ensures atomicity and consistency.

## Concurrency

### Graceful Shutdown

The server (`cmd/server/main.go`) listens for `SIGINT` or `SIGTERM` signals. When a shutdown signal is received:

1. The server stops accepting new connections.
2. It waits up to **10 seconds** for existing requests to complete.
3. After the timeout, any remaining connections are forcibly closed and the program exits.

This prevents abrupt termination that could leave orders in an inconsistent state.

### Background Tasks

Long‑running or non‑critical tasks (email sending, logging) are executed in goroutines to keep HTTP response times low. The goroutines are **fire‑and‑forget**; errors are logged but do not affect the primary request.

## Middleware Stack

The Chi router is configured with a standard middleware stack (see `handler.go`):

1. **Logger**: Logs each request’s method, path, and duration.
2. **Recoverer**: Recovers from panics, returns a `500 Internal Server Error`, and prevents the server from crashing.
3. **CORS**: Configures Cross‑Origin Resource Sharing based on `ALLOWED_ORIGINS`. Supports credentials and pre‑flight caching.
4. **AdminOnly**: Custom middleware that validates JWT tokens for protected admin routes.

## Configuration

The `config` package (`internal/config/config.go`) loads environment variables and validates them at startup. Required variables are listed in `.env‑example`.

```go
type Config struct {
    Port            string   `env:"PORT,required"`
    AllowedOrigins  []string `env:"ALLOWED_ORIGINS,required"`
    DBURL           string   `env:"DB_URL,required"`
    JWTSecret       string   `env:"JWT_SECRET,required"`
    ResendApiKey    string   `env:"RESEND_API_KEY,required"`
    OrderEmailTo    string   `env:"ORDER_EMAIL_TO,required"`
    OrderEmailFrom  string   `env:"ORDER_EMAIL_FROM,required"`
}
```

If a required variable is missing, the application fails fast with a clear error message.

## Logging

The backend uses Go’s **structured logging** via `slog`. The default logger is configured to output JSON in production and human‑readable text in development.

```go
log.Printf("CreateOrder: failed to build snapshots: %v", err)
```

All logs include timestamps and source file/line information. In a production environment, these logs can be ingested by tools like **Loki**, **ELK**, or cloud logging services.

## Error Handling

HTTP handlers follow a uniform error‑response pattern via the `respond.go` helpers:

- `respondBadRequest(w, message)` → `400` with `{"error": message}`
- `respondUnauthorized(w, message)` → `401`
- `respondInternalError(w)` → `500` with a generic message (to avoid leaking internal details)

Errors that are not expected (e.g., database connection failures) are logged and result in a `500` response.

## Testing

The backend includes both **unit tests** and **integration tests**.

- **Unit Tests**: Test individual functions in isolation (e.g., validation logic). Located in `*_test.go` files next to the code they test.
- **Integration Tests**: Test the full HTTP stack against a real PostgreSQL database. These tests are in `internal/handler/integration_*.go` and are run only when the `INTEGRATION` environment variable is set.

**Running Tests**:

```bash
go test ./...                     # run unit tests
INTEGRATION=1 go test ./...       # run integration tests (requires DB)
```

## Performance Considerations

- **Connection Pooling**: `pgx` maintains a pool of database connections; the pool size is configured via `DB_URL` parameters.
- **No N+1 Queries**: The menu endpoint performs one query per category to fetch items. This is acceptable for a small menu but could be optimized with a single join if the menu grows.
- **Statelessness**: The API is stateless; any instance can handle any request. This allows horizontal scaling behind a load balancer.

## Security

- **JWT Authentication**: Described in detail in the [Authentication](../docs/AUTHENTICATION.md) document.
- **Input Sanitization**: All user input is validated; SQL queries are parameterized (via `sqlc`) to prevent injection.
- **CORS Restrictions**: Only whitelisted origins are allowed.

## Future Enhancements

- **Rate Limiting**: Add a rate‑limiting middleware (e.g., using `go‑rate‑limiter`) for public endpoints.
- **Health Checks**: Extend the `/health` endpoint to include database connectivity and external service status.
- **Metrics**: Expose Prometheus metrics (request duration, error counts) for monitoring.
- **Event‑Driven Architecture**: Replace direct email calls with an internal message queue (e.g., NATS) for better decoupling.
